# Data Rights Lifecycle

LEGAL-02 implements the personal-account data export and account deletion lifecycle. Requests require a session issued within 15 minutes, exact handle confirmation, and the current account version. A subject is persisted only as a one-way `subject_` hash in operational evidence.

## Owner Flow

`POST /api/users/me/data-rights/requests` creates either `data_export` or `account_deletion`. Owners can list and read only their own requests. Eligible requests can be cancelled with their current request version. No more than three requests may be created in a rolling 30-day window, and duplicate active requests of the same type are rejected.

An export is generated by an authorized operator, is capped at 5 MiB, expires after seven days, and yields a private download valid for exactly 15 minutes. The package covers account, profile, task, community, library, media, generation, decrypted Chat, notification, support, and billing data. Credential, token, secret, ciphertext, raw payload, cookie, authorization, and network-hash fields are recursively removed.

## Deletion Flow

Account deletion has a 30-day cancellation window. Primary processing revokes identity, sessions, developer credentials, and webhooks; deletes Chat and notifications; marks media for object cleanup; anonymizes profile, creative, community, and task content; and retains only minimum support, billing, audit, and safety facts. The transaction emits one immutable receipt for each of 15 declared domains.

Primary deletion changes the request to `primary_completed`. Final completion requires immutable expiry receipts for `primary_database`, `object_storage`, and `audit_archive`, each recorded no earlier than 35 days after primary completion. Ordinary updates and deletes on event, export, deletion, and backup evidence tables are rejected by PostgreSQL triggers. Maintenance cleanup requires the explicit transaction-local setting `app.data_rights_maintenance=on`.

## Operations

The Admin Users workspace exposes queue filters, aggregate active/completed/overdue metrics, request processing, and backup evidence entry. `admin:data-rights:read` grants queue access; `admin:data-rights:manage` grants primary processing and backup receipt creation. Every Admin mutation is domain audited and uses optimistic request versions.

Do not mark production backup expiry complete from fixture or local evidence. Production acceptance still requires RELEASE-01 infrastructure, an actual backup inventory, expiry execution, restore-negative verification, and retained audit evidence. The governance status deliberately records export/deletion engineering as implemented while `backupDeletionRehearsed` remains false.
